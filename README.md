### рҹ”Қ What We Audited (Every Component)

| Component | Source | Outbound calls | Verdict |
|-----------|--------|:---:|:---:|
| **server.py** (ours) | We wrote it line by line | **0** | вң… Safe |
| **browser agent** (separate repo) | [nicedreamzapp/browser-agent](https://github.com/nicedreamzapp/browser-agent) вҖ” we wrote it | **0** (talks to localhost CDP only) | вң… Safe |
| **mlx-lm** | Apple ML team | **0** | вң… Safe |
| **MLX framework** | Apple | **0** | вң… Safe |
| **Model weights** | HuggingFace verified mlx-community repos | **0** at runtime | вң… Safe |
| **iMessage scripts** | Pure shell + AppleScript | localhost only (Studio Record port 17494) | вң… Safe |
| **Claude Code CLI** | Anthropic (closed-source binary) | **1 non-blocking** startup call to `api.anthropic.com` вҖ” inference still stays 100% local even if the call is firewalled | вҡ пёҸ Disclosed |

> в„№пёҸ **On that one exception:** Claude Code's own binary attempts a startup handshake to `api.anthropic.com` (likely version/session check). We can't suppress it вҖ” it's baked into Anthropic's closed-source CLI. **It's non-blocking**: block it at the firewall and Claude Code still works fine with your local model. Your prompts, code, and completions never leave the machine. Verified with `lsof -i -P` once the model is loaded.

### вҡ пёҸ Transparency Note: Claude Code's Own Binary

Claude Code is Anthropic's closed-source CLI. On startup, it may attempt a **non-blocking** connection to `api.anthropic.com` (likely session validation or version check) вҖ” even when `ANTHROPIC_BASE_URL` points to localhost. **This does not affect your code or prompts** вҖ” all inference still goes to the local MLX model. If you're offline or a firewall blocks the call, Claude Code still works fine.

Our code (server.py, launchers, scripts) makes **zero** outbound connections. But we can't suppress the startup behavior of a closed-source binary we don't control. If full network isolation matters to you, use a firewall rule to block `api.anthropic.com` вҖ” everything will still work.

### рҹҡ« What We Ripped Out

> вҡ пёҸ We **[removed LiteLLM](https://x.com/Tahseen_Rahman/status/2035501506242240520)** after supply-chain attack concerns. Every dependency was re-audited from scratch. If a package had unexplained network calls, it didn't ship.

### вң… What This Means in Practice

| Scenario | Cloud Claude | This Repo |
|---|:---:|:---:|
| Working with NDA / proprietary code | вқҢ Risky | вң… Air-gapped |
| Coding on a plane (no wifi) | вқҢ Doesn't work | вң… Works |
| Running on a kill-switch firewall | вқҢ Blocked | вң… Works |

> рҹ”’ **The math is simple:** if there are no outbound HTTP calls, your data cannot leak. We grep'd every file for `requests`, `urllib`, `urlopen`, `httpx`, `socket.connect` вҖ” the only network calls in the entire codebase are to `localhost`. Run `lsof -i -P` while it's running. You'll see nothing leaving your Mac.

## рҹ”§ Tool-Call Reliability (v2 вҖ” March 2026)

Local models don't format tool calls perfectly. They *want* to call a tool but mix XML and JSON syntax. Claude Code sees no valid tool call, re-prompts, and the model does it again. The result: **infinite loops where the AI says "let me do that" but never actually does anything.**

We fixed this. Here's what was happening and what we did about it.

### рҹҗӣ The Problem

The model was generating garbled tool calls like this:
```
<tool_call>
<function=Bash><parameter=command>rm -rf /tmp/old</parameter></function>
</tool_call>
```

Instead of the correct JSON format Claude Code expects:
```json
<tool_call>
{"name": "Bash", "arguments": {"command": "rm -rf /tmp/old"}}
</tool_call>
```

The JSON parser choked, Claude Code saw no tool call, re-prompted the model, and the model garbled it the exact same way again вҖ” creating an infinite loop.

### вң… The Fix (4 changes to `server.py`)

| Change | What | Why |
|--------|------|-----|
| **KV Cache** | 4-bit вҶ’ 8-bit, quantization starts at token 1024 | Model retains conversation context instead of "forgetting" earlier messages |
| **Temperature** | 0.7 вҶ’ 0.2 | Less randomness = more consistent tool formatting |
| **Garbled Recovery** | New `recover_garbled_tool_json()` function | Catches XML-in-JSON hybrids, `<function=X><parameter=Y>` inside `<tool_call>` tags, and infers tool names from parameter keys |
| **Retry Logic** | Up to 2 retries when tool intent is detected but parsing fails | Re-prompts with explicit formatting instructions before giving up |

### рҹ§Ә Test Results

We built an automated test suite (`scripts/test_mlx_server.py`) that sends real Anthropic API requests to the server simulating multi-step tasks вҖ” the exact kind that were failing before.

The server (`proxy/server.py`) is **one file, ~1000 lines**. It does six things:

1. рҹ“Ұ **Loads the model** вҖ” Apple's MLX framework, native Metal GPU, unified memory. Handles Gemma's `RotatingKVCache` quirk automatically so sliding-window models don't crash on the first request.
2. рҹ”Ң **Speaks Anthropic API** вҖ” Claude Code thinks it's talking to Anthropic's cloud. It's not.
3. рҹ”§ **Translates tool use** вҖ” Three different tool-call formats in and out: Gemma 4 native (`<|tool_call>call:Name{...}<tool_call|>`), Llama 3.3 raw JSON (`{"type":"function",...}`), and HuggingFace `<tool_call>` JSON (Qwen and others). All converted вҶ” Anthropic `tool_use` blocks, with garbled-output recovery for small models.
4. рҹ§№ **Cleans the output** вҖ” Local models think out loud in `<think>` / `<|channel>thought` tags, emit stop markers (`<turn|>`, `<|python_tag|>`), and sometimes drop in reasoning preamble. We strip all of it before sending back to Claude Code.
5. вҡЎ **Reuses prompt caches across requests** вҖ” so Claude Code's 4K-token system prompt doesn't get re-prefilled on every turn. Huge speedup for short questions.
6. рҹҺҜ **Code mode** вҖ” auto-detects Claude Code coding sessions (any of Bash/Read/Edit/Write/Grep/Glob in the tools list) and swaps Claude Code's ~10K-token harness prompt for a slim ~100-token one tuned for local models. Cuts prompt tokens by 99% and stops models from refusing with "I am not able to execute this task."

### What makes this actually work

- рҹҺҷпёҸ **Speech-in** вҖ” a compiled Swift binary wraps Apple's `SFSpeechRecognizer` (the same on-device engine that powers macOS Dictation) in a *continuous* listening loop rather than the usual Fn-Fn toggle. End of utterance is detected via **partial-result stability**: if the transcribed text stops changing for 2.5 seconds, the recognizer finalizes that sentence. That's way more robust than silence/RMS heuristics against background noise, fans, or music.
- рҹ”Ҡ **Speech-out** вҖ” a CLI at `~/.local/bin/speak` wraps **Pocket TTS** driving a cloned copy of Matt's own voice. Any TTS that accepts a string and plays audio slots in вҖ” macOS `say`, Piper, local ElevenLabs, your choice.
- рҹ”Ғ **Feedback-loop prevention** вҖ” the listener auto-pauses while `afplay` is running, so the TTS output of one turn never gets picked up as input for the next. No "the model talking to itself" loops.
- рҹ§  **Speak-every-turn is enforced via system prompt** вҖ” `NarrativeGemma/CLAUDE.md` is loaded as the narration persona. It tells Gemma to narrate every tool call, every reasoning step, every result, *before* it writes the text reply. You're never staring at a silent terminal wondering if it's thinking.
- рҹӣЎпёҸ **Real production hardening** вҖ” 10-minute preventive process recycle (dodges a known `SFSpeech` daemon wedge), queue-backlog detection with a non-zero exit code when the listener is stuck. Not a demo script вҖ” a tool that has to run unattended for hours.

### Why it matters

"Voice-controlled AI" is everywhere right now, but under the hood almost every public demo is a cloud pipeline wearing a local-looking coat. If the network drops, the demo dies. If your client's laptop blocks outbound connections, the demo dies. If you're on a plane, in a Faraday cage, or debugging on a disconnected-by-policy machine, the demo dies.

This setup doesn't die. **Apple's on-device speech engine is a fully local model that already ships with the OS**, and accessing it via `SFSpeechRecognizer` is a first-class macOS API вҖ” it's just that almost nobody wraps it in a continuous-listen daemon with production hardening and plumbs it to a local LLM with a cloned-voice reply stream. Now there's one.

### How to wire it up

> рҹӣ пёҸ **The listening stack lives in its own repo.** The `Listen.swift` binary, the `dictation` / `dispatch` / `inject` scripts, and the `narrative-claude.sh` launcher are **a sibling project**: [`nicedreamzapp/NarrateClaude`](https://github.com/nicedreamzapp/NarrateClaude). ### The two halves of the loop, and where each half lives

**рҹ—ЈпёҸ The speak-and-think half (this repo, `claude-code-local`):**
- `launchers/Narrative Gemma.command` вҖ” boots the MLX server with Gemma 4 31B and injects the narration persona via `MLX_APPEND_SYSTEM_PROMPT_FILE` so Gemma narrates every turn

- An AppleScript injector that writes transcribed utterances straight into the bound Terminal tab by window ID
- A `narrative-claude.sh` one-click launcher that opens the Terminal, starts Claude Code, captures the window ID, and starts the listener

## рҹӣӨпёҸ The Journey

We didn't start here. We went through three generations in one night:

| Gen | What We Tried | Speed | рҹ’Ў What We Learned |
|:---:|---|:---:|---|
| 1пёҸвғЈ | Ollama + custom proxy | 30 tok/s | Ollama works but Claude Code can't talk to it directly |
| 2пёҸвғЈ | llama.cpp TurboQuant + proxy | 41 tok/s | TurboQuant compresses KV cache 4.9x, but the proxy is the bottleneck |
| 3пёҸвғЈ | **MLX native server** | **65 tok/s** | **Kill the proxy. Speak Anthropic API directly. 7.5x faster.** |
| 4пёҸвғЈ | **The lineup** | 65 / 15 / 7 tok/s | Three brains, one server. Same MLX, same Anthropic API вҖ” swap one env var to change the fighter. |

> рҹҺҜ Each generation taught us something. Killing the proxy made it fast. Adding the lineup made it flexible.

### рҹӘҙ Why this matters вҖ” the ambient-computing angle

The real goal isn't "a faster Claude Code" вҖ” it's **getting off screens and mice**. Hunched-over-screen computing is breaking our bodies: carpal tunnel, curved spines, $1500 ergonomic chairs bought to patch the damage the rest of the desk is doing. That era is ending. These three repos are pieces of what comes next вҖ” **computing that's around you instead of in front of you**. Screens become optional, typing becomes optional, sitting still becomes optional, but your data and your voice never leave your house.
